## 2.2 OS Dashboard — 1단계 운영 구조

### 이벤트 기반 구조

- 모든 상태 변경은 dispatch → reducer → history append 흐름을 따른다.
    
- history.json은 append-only 구조를 유지한다.
    
- 앱 시작 시 history replay를 통해 상태를 복원한다.
    
- reducer가 모든 상태 변경을 단일 경로로 처리한다.
    

### Goal / Track 구조

- Goal은 전역 Map 구조로 관리된다.
    
- Track은 전략 단위이며 Goal을 소유하지 않는다.
    
- Goal은 trackId를 통해 Track에 연결된다.
    
- Track 진행률은 Goal의 done 개수 기반으로 자동 계산된다.
    

### 이벤트 타입 (Goal 관련)

- GOAL_CREATED
    
- GOAL_UPDATED
    
- GOAL_STATUS_CHANGED
    
- GOAL_DELETED

# 6️⃣ 실행 모드 레이어 (팀 고정 규칙)

## ⚙️ EXECUTION MODE — TOKEN MIN 기본 적용 규칙 (v3.6 추가 블록)

목적:  
대형 코드 파일 수정 시 불필요한 토큰 소비를 방지하고,  
1단계 시스템 안정화 범위 내에서 구조 안전성을 유지하기 위한 실행 규칙을 정의한다.

### 기본 적용 원칙

운영팀_개발이 수행하는 모든 코드 수정 작업은  
기본적으로 TOKEN MIN MODE를 따른다.

사용자가 다음과 같이 지시할 경우:

"요청 텍스트 작성해"

자동으로 TOKEN MIN MODE 템플릿을 적용한다.

### TOKEN MIN MODE 정의

1. 전체 파일 재출력 금지
    
2. 전체 파일 다중 분할 읽기 금지
    
3. 수정 대상 블록만 확인
    
4. diff 단위 패치만 작성
    
5. 미변경 코드 재출력 금지
    
6. 설명 최소화 (핵심만)
    

### 패치 방식

모든 수정은 AG_CODE_PATCH_TEMPLATE_v1 방식을 따른다.  
전면 리팩토링은 1단계 시스템 안정화 범위에서 금지한다.

### 적용 대상

- App.jsx 수정
    
- reducer 수정
    
- Snapshot 엔진 수정
    
- 자동화 스크립트 수정
    
- 대형 단일 파일 수정 작업
